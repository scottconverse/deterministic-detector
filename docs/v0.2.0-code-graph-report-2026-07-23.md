## Traps found and encoded in the instructions

1. **`tests_for` under-reports ~20×** — it matches test *names*, not call
   edges (1 of 20 on the acceptance target). Dependent tests come from
   `callers_of` filtered to `is_test` nodes. Encoded in instructions.md and
   the installer skill.
2. **Ambiguous bare names** return a candidate list; re-query with
   `qualified_name`. Deterministic, correct behavior — documented.
3. **`.git` marker required** at the root the tool indexes (non-issue for
   real repos).
4. **Young project** — version pinned everywhere it's invoked; any bump
   re-runs the (cheap) validation spike first.

## What shipped

- `.mcp.json` — `code-graph` server via
  `uvx --from code-review-graph==2.3.7 code-review-graph serve`, tool-filtered
  to the 5 tools the harness needs (of 30) to keep session context small:
  `query_graph_tool, get_impact_radius_tool, detect_changes_tool,
  build_or_update_graph_tool, list_graph_stats_tool`.
- Installer step 5: per-repo `build` + `.gitignore` entry + one-real-query
  acceptance + re-index guidance.
- instructions.md: scoped blast-radius rule — non-trivial changes only
  (trivial changes never require a graph query, per D7's rejection of the
  original spec's always-on rule); silent skip where no index exists.
- Version 0.2.0. The detector repo itself is indexed (dogfooding);
  `.code-review-graph/` gitignored.
